# 애노테이션 트랜잭션 속성과 포인트 컷

세밀한 트랜잭션 속성의 제어가 필요한 경우에는 메소드 이름 패턴을 이용해서 일괄적으로 트랜잭션 속성을 부여하는 방식은 적합하지 않음.

why → 기본 속성과 다른 경우가 있을 때마다 일일이 포인트 컷과 어드바이스를 새로 추가해주어야 하여 포인트 컷 자체가 지저분해지고 설정 파일도 복잡해지기 쉽기 때문.

이런 경우에는 스프링이 제공하는 직접 타깃에 트랜잭션 속성 정보를 가진 애노테이션을 지정하는 방법이 적합.

## 트랜잭션 애노테이션

타깃에 부여할 수 있는 트랜잭션 애노테이션들.

- Transactional - 타깃이 메소드와 타입으로 메소드, 클래스, 인터페이스에 사용 가능. 기본적으로 트랜잭션 속성을 정의하기 위함이지만, 동시에 포인트 컷의 자동 등록에도 사용.

### 트랜잭션 속성을 이용하는 포인트 컷

<img width="581" height="152" alt="image" src="https://github.com/user-attachments/assets/e66b4b87-2e0c-4a11-bf7f-d0cd36ef7f53" />

위 이미지는 @Transactional 애노테이션을 사용했을 때 어드바이저의 동작 방식.

TransactionInterceptor는 메소드 이름 패턴을 통해 부여되는 일괄적인 트랜잭션 속성 정보 대신 @Transactional 애노테이션의 엘리먼트에서 트랜잭션 속성을 가져오는 AnnotationTransactionAttributeSource를 사용.

@Transactional은 메소드마다 다르게 설정할 수도 있으므로 매우 유연한 트랜잭션 속성 설정이 가능해짐.

그런데 이렇게 하면 유연한 속성 제어는 가능하겠지만 코드가 지저분해지고 동일한 속성 정보를 가진 애노테이션을 반복적으로 메소드마다 부여해주는 바람직한 못한 결과 발생 가능.

### 대체 정책

그로 인해 스프링에서는 @Transactional을 적용할 때 4단계의 대체 정책을 이용하게 해줌.

메소드의 속성을 확인할 때 타깃 메소드, 타깃 클래스, 선언 메소드, 선언 타입(클래스, 인터페이스)의 순서에 따라 @Transactional이 적용됐는지 차례로 확인하고, 가장 먼저 발견되는 속성 정보를 사용하게 하는 방법.

끝까지 발견되지 않으면 해당 메소드는 트랜잭션 적용 대상이 아니라고 판단.

<img width="554" height="389" alt="image" src="https://github.com/user-attachments/assets/148ca076-270c-4752-ba9e-520d488f2442" />

위와 같이 정의된 인터페이스와 구현 클래스가 있는 경우 인터페이스에 두 개의 메소드가 있고, 구현 클래스 역시 인터페이스의 정의된 두 개의 메소드를 갖고 있음.

구현 클래스인 ServiceImpl이 빈으로 등록됐고 그 두 개의 메소드가 트랜잭션의 적용 대상이 되어야 한다면 @Transactional을 부여할 수 있는 위치는 총 6개.

@Transactional이 위치할 수 잇는 첫 번째 후보는 [5], [6].

왜냐하면 스프링은 타깃 오브젝트의 메소드부터 @Transactional 애노테이션이 존재하는지 확인하기 때문.

메소드에서 @Transactinoal을 발견하지 못하면, 다음은 타깃 클래스인 [4]에서 확인.

해당 클래스에서 발견되는 경우 클래스 내의 모든 메소드에 공통적으로 적용되는 속성이 됨.

타깃 클래스에서도 @Transactional을 바견하지 못하면, 인터페이스 내의 메소드인 [2], [3]에서 확인 후 속성 적용.

인터페이스 메소드에도 없는 경우 마지막으로 인테퍼이스 타입 [1]의 위치를 확인.

@Transactional은 먼저 타입 레벨에 정의되고 공통 속성을 따르지 않는 메소드에 대해서만 메소드 레벨에 다시 @Transactional을 부여해주는 식으로 사용해야 함.

기본적으로 @Transactional 적용 대상은 클라이언트가 사용하는 인터페이스가 정의한 메소드이므로 @Transactional도 타깃 클래스보다는 인터페이스에 두는 게 바람직함.

하지만 인터페이스를 사용하는 프록시 방식의 AOP가 아닌 방식으로 트랜잭션을 적용하면 인터페이스에 정의한 @Transactional은 무시되기 때문에 안전하게 타깃 클래스에 @Transactional을 두는 방법을 권장.

인터페이스에 @Transactional을 두면 구현 클래스가 바뀌더라도 트랜잭션 속성을 유지할 수 있다는 장점이 존재.

### 트랜잭션 애노테이션 사용을 위한 설정

<tx:annotation-driven /> : 해당 태그 사용 시 트랜잭션 애노테이션을 이용하는 데 필요한 어드바이저, 어드바이스, 포인트컷, 애노테이션을 이용하는 틀내잭션 속성 정보가 등록됨.

## 트랜잭션 애노테이션 적용

@Transactional은 세밀한 트랜잭션이 필요할 때만 사용하는 것은 아님.

트랜잭션 설정이 직관적이고 간단하다고 생각해 사용하는 경우도 많음.

다만 트랜잭션 적용 대상을 손쉽게 파악할 수 없어 트랜잭션이 적용되지 않았다는 사실은 파악하기 쉽지 않음.

따라서 @Transactional을 사용할 때는 주의가 필요하며 적용에 대한 별도의 코드 리뷰를 거칠 필요가 있지만 일부 데이터 액세스 기술은 트랜잭션이 시작되지 않으면 아예 DAO에서 예외가 발생하기도 함.

JDBC를 직접 사용하는 기술의 경우는 트랜잭션이 없어도 DAO가 동작할 수 있기에 주의.

<img width="710" height="186" alt="image" src="https://github.com/user-attachments/assets/0cd44adb-91ec-4824-845e-a647a8e4399a" />

<img width="707" height="418" alt="image" src="https://github.com/user-attachments/assets/acfa2b95-eb58-4a02-87ea-66dd33d556ff" />

위는 <tx:attributes> 태그를 이용해 설정했던 설정을 애노테이션으로 바꾼 이미지.

애노테이션 이용 시에는 두 가지 속성 중 가장 많이 사용되는 한 가지를 타입 레벨에 공통 속성으로 지정해주고, 나머지 속성은 개별 메소드에 적용해야 함.

메소드 레벨의 속성은 메소드마다 반복돼아 하므로 속성의 종류가 두 가지 이상이고 적용 대상 메소드의 비율이 비슷하다면 메소드에 많은 @Transactional이 반복될 수 있음.

애노테이션을 이용한 트랜잭션 속성 지정은 tx 스키마를 사용할 때와 마찬가지로 IDE의 자동 완성 기능을 활용할 수 있고 속성을 잘못 지정한 경우 컴파일 에러가 발생해서 손쉽게 확인 가능하다는 장점이 존재.

# 트랜잭션 지원 테스트

## 선언적 트랜잭션과 트랜잭션 전파 속성

트랜잭션을 정의할 떄 지정할 수 있는 트랜잭션 전파 속성은 매우 유용한 개념.

예를 들어 REQUIRED로 전파 속성을 지정해줄 경우, 앞에서 진행 중인 트랜잭션이 있으면 참여하고 없으면 자동으로 새로운 트랜잭션을 시작해줌.

트랜잭션 적용 때문에 불필요하게 코드를 중복하는 것도 피할 수 있으며, 애플리케이션을 작은 기능 단위로 쪼개서 개발 가능.

작업 단위가 다른 비즈니스 로직이 있는 경우 독자적인 트랜잭션 단위 또는 다른 트랜잭션의 일부로 참여 가능.

만일 트랜잭션의 전파 방식을 이용 할 수 없었다면 트랜잭션의 중복이 일어나기 시작하고 이를 관리하는 불편이 뒤따르게 됨.

<img width="599" height="241" alt="image" src="https://github.com/user-attachments/assets/4c18142e-4b81-4645-b1b4-0ba8713ad149" />

AOP 를 이용해 코드 외부에서 트랜잭션의 기능을 부여해주고 속성을 지정할 수 있게 하느 ㄴ방법을 선언적 트랜잭션(declarative transaction)이라고 함.

반대로 TransactionTemplate이나 개별 데이터 기술의 트랜잭션 API를 사용해 직접 코드 안에서 사용하는 방법은 프로그램에 의한 트랜잭션(programmatic transaction).

특별한 경우가 아니라면 선언적 방식의 트랜잭션을 사용하는 것이 바람직.

## 트랜잭션 동기화와 테스트

AOP 덕분에 프록시를 이용한 트랜잭션 부가 기능을 간단하게 애플리케이션 전반에 적용 가능.

그리고 트랜잭션 추상화 덕분에 데이터 액세스 기술에 상관없이, 또 트랜잭션 기술에 상관없이 DAO에서 일어나는 작업들을 하나의 트랜잭션으로 묶어서 추상레벨에서 관리 가능.

이를 통해 AOP를 통한 선언적 트랜잭션이나 트랜잭션 전파 등은 불가능했을 것.

### 트랜잭션 매니저와 트랜잭션 동기화

트랜잭션 추상화 기술의 핵심은 트랜잭션 매니저와 트랜잭션 동기화.

PlatformTransactionManager 인터페이스를 구현한 트랜잭션 매니저를 통해 구체적인 트랜잭션 기술의 종류에 상관없이 일관된 트랜잭션 제어가 가능했고, 트랜잭션 동기화 기술이 있었기에 시작된 트랜잭션 정보를 저장소에 보관해뒀다가 DAO에 공유 가능했음.

트랜잭션 동기화 기술은 트랜잭션 전파를 위해서도 중요한 역할을 함.

진행 중인 트랜잭션이 있는지 확인하고, 트랜잭션 전파 속성에 따라서 이에 참여할 수 있도록 만들어주는 것도 트랜잭션 동기화 기술 덕분.

특별한 이유가 있다면 트랜잭션 매니저를 이용해 트랜잭션에 참여하거나 트랜잭션을 제어하는 방법을 사용 가능.

### 트랜잭션 매니저를 이용한 테스트용 트랜잭션 제어

각 메소드의 트랜잭션 전파 속서이 REQUIRE이면 이 메소드들이 호출되기 전에 트랜잭션이 시작되게만 한다면 트랜잭션 매니저를 이용한 트랜잭션 제어가 가능.

그런데 메소드를 추가하지 않고도 테스트 코드만으로 세 메소드의 트랜잭션을 통합하는 방법이 존재.

테스트 메소드에서 UserService의 메소드를 호출하기 전에 트랜잭션을 미리 시작하면 됨.

트랜잭션의 전파는 트랜잭션 매니저를 통해 트랜잭션 동기화 방식이 적용되기 때문에 가능.

그렇다면 테스트에서 트랜잭션 매니저를 이용해 트랜잭션을 시작시키고 이를 동기화해주면 됨.

<img width="711" height="387" alt="image" src="https://github.com/user-attachments/assets/23f50136-c4a4-45fc-942c-95cfc6ed15bd" />

트랜잭션을 시작하기위해서 먼저 트랜잭션 정의를 담은 오브젝트를 만들고 이를 트랜잭션 매니저에 제공하면서 새로운 트랜잭션을 요청하면 됨.

트랜잭션 매니저는 이미 @Autowired를 테스트 코드로 주입하게 설정.

이렇게 하면 테스트 코드에서 트랜잭션 매니저를 이용해서 트랜잭션을 만들고 그 후에 실행되는 UserService의 메소드들이 같은 트랜잭션에 참여하게 만들 수 있음.

### 트랜잭션 동기화 검증

트랜잭션 속성 중 읽기 전용과 제한 시간 등은 처음 트랜잭션이 시작할 때만 적용되고 그 이후에 참여하는 메소드의 속성은 무시됨.

즉 deleteAll()의 트랜잭션 속성은 쓰기 가능으로 되어 있지만 앞에서 시작된 트랜잭션이 읽기 전용이라고 하면 deleteAll()의 모든 작업도 읽기 전용 트랜잭션이 적용된 상태에서 진행된다는 말.

<img width="707" height="286" alt="image" src="https://github.com/user-attachments/assets/d4116ffd-2fdf-4bf7-8c47-8fb24387a2ae" />

트랜잭션 속성을 강제로 읽기 전용으로 만들고 테스트해보면 예외 발생으로 인해 실패할 것.

예외는 TransientDataAccessResourceException이고 메시지는 Connection is read-only라고 나올 것.

읽기 전용 트랜잭션에 대해 쓰기 작업을 하는 경우 발생하는 예외.

위 에러를 통해 deleteAll()이 테스트 코드 내에서 시작한 트랜잭션에 참여하고 있음을 확인 가능.

테스트를 통해 확인할 수 있듯이 스프링의 트랜잭션 추상화가 제공하는 트랜잭션 동기화 기술과 트랜잭션 전파 속성 덕분에 테스트도 트랜잭션으로 묶을 수 있음.

이런 방법은 선언적 트랜잭션이 적용된 서비스 메소드에만 적용되는 것이 아니고 JdbcTemplate과 같이 스프링이 제공하는 데이터 액세스 추상화를 적용한 DAO에도 동일한 영향을 미침.

JdbcTemplate은 트랜잭션이 시작된 것이 잇으면 그 트랜잭션에 자동으로 참여하고, 없으면 트랜잭션 없이 자동 커밋 모드로 JDBC 작업을 수행 → 마치 REQUIRED인 것처럼.

<img width="709" height="489" alt="image" src="https://github.com/user-attachments/assets/52c07768-ec1d-43be-a58b-a1e20a360f21" />

테스트에서 트랜잭션을 시작하거나 조작할 수 있는 기능은 매우 유용.

테스트 코드에서 미리 트랜잭션을 시작해놓으면 직접 호출하는 DAO 메소드도 하나의 트랜잭션으로 묶을 수 있음.

트랜잭션 결과나 상태를 조작하면서 테스트도 가능.

또한 테스트 메소드 안에서 트랜잭션을 여러 번 만들거나 트랜잭션 속성에 따라 여러 메소드를 조합해 사용할 때 어떤 결과가 나오는 지도 미리 검증 가능.

### 롤백 테스트

롤백 테스트는 테스트 코드로 트랜잭션을 제어해서 적용할 수 있는 테스트 기법.

테스트 내의 모든 DB 작업을 하나의 트랜잭션 안에서 동작하게 하고 테스트가 끝나면 무조건 롤백해버리는 테스트.

아래 이미지는 전형적인 롤백 테스트.

<img width="710" height="437" alt="image" src="https://github.com/user-attachments/assets/fc736adf-9548-49c6-ba36-3d58a0c844b9" />

롤백 테스트는 DB 작업이 포함된 테스트가 수행돼도 DB에 영향을 주지 않기에 장점이 많음.

테스트에서 DB에 쓰기 작업 기능을 실행하면서 테스트를 수행하면 DB의 데이터가 바뀜.

결국 DB를 액세스하는 테스트를 위해서는 테스트를 할 때마다 테스트 데이터를 초기화하는 번거로운 작업이 필요해짐.

하지만 롤백 테스트를 이용하면 테스트를 진행하는 동안 조작한 데이터를 모두 롤백하고 테스트를 시작하기 전 상태로 만들어주기 때문에 유용함.

심지어 여러 개발자가 하나의 공용 테스트용 DB를 사용할 수 있게도 해줌.

적절한 격리 수준만 보장해주면 동시에 여러 개의 테스트가 진행돼도 상관 X.

DB에 따라서 성공적인 작업이라도 트랜잭션을 롤백하면 커밋할 때보다 성능이 더 향상되기도 함.

물론 DB의 트랜잭션 처리 방법에 따라 롤백이 커밋보다 더 많은 부하를 주는 경우도 있으니 단지 성능 때문에 롤백 테스트가 낫다는 것은 아님.

## 테스트를 위한 트랜잭션 애노테이션

### @Transactional

테스트에도 @Transactional을 적용 가능.

테스트 클래스 또는 메소드에 @Transactional 애노테이션을 부여해주면 마치 타깃 클래스나 인터페이스에 적용된 것처럼 테스트 메소드에 트랜잭션 경계가 자동으로 설정됨.

이를 이용하면 테스트 내에서 진행하는 모든 트랜잭션 관련 작업을 하나로 묶어줄 수 있음.

@Transactional에는 모든 종류의 트랜잭션 속성을 지정할 수 있기도 함.

물론 테스트에서 사용하는 @Transactional은 AOP를 위한 것은 아님.

단지 컨텍스트 테스트 프레임워크에 의해 트랜잭션을 부여해주는 용도로 쓰일 뿐.

<img width="705" height="245" alt="image" src="https://github.com/user-attachments/assets/3951cd74-902c-4e9b-8536-8227105546b0" />

테스트의 트랜잭션 속성을 메소드 단위에 부여한 이미지.

테스트 메소드 실행 전에 새로운 트랜잭션을 만들어주고 메소드가 종료되면 트랜잭션을 종료해줌.

<img width="715" height="179" alt="image" src="https://github.com/user-attachments/assets/47241c78-8587-4470-8f6c-3f715cc136c1" />

트랜잭션 적용 여부 확인은 위와 같이 읽기 전용으로 바꾸고 테스트를 실행해 예외 발생을 확인해보면 됨.

@Transactional은 테스트 클래스 레벨에도 부여 가능.

그러면 테스트 클래스 내의 모든 메소드에 트랜잭션이 적용됨.

### @Rollback

테스트 메소드나 클래스에 사용하는 @Transactional는 애플리케이션의 클래스에 적용할 때와 디폴트 속성은 동일하지만 테스트가 끝나면 자동으로 롤백된다는 점이 다름.

테스트에 적용된 @Transactional은 기본적으로 트랜잭션을 강제 롤백시키도록 설정되어 있음.

하지만 테스트 메소드 내의 작업을 하나의 트랜잭션으로 묶고 싶지만 강제 롤백을 원하지 않는 경우에는 @Rollback을 사용하면 됨.

@Rollback의 기본 값은 true이고 롤백을 원치 않는 경우에는 @Rollback(false)을 하면 됨.

<img width="712" height="227" alt="image" src="https://github.com/user-attachments/assets/ce62d125-e626-4e31-98f5-b4e634100f85" />

위와 같이 테스트를 진행하면 예외가 발생하지 않는 한 트랜잭션이 커밋됨.

### @TransactionConfiguration

테스트 클래스의 모든 메소드에 트랜잭션을 적용하면서 모든 트랜잭션이 롤백되지 않고 커밋되게 하려면 모든 메소드에 @Rollback(false)를 할 수도 있지만 그보다는 클래스 레벨에 부여할 수 있는 @TransactionConfiguration 애노테이션을 이용하면 편리.

<img width="711" height="304" alt="image" src="https://github.com/user-attachments/assets/332d18b5-d7f3-40dc-a178-d140b0424688" />

@TransactionConfiguration을 사용하면 롤백에 대한 공통 속성 지정 가능.

디폴트 롤백 송성은 false로 해두고 테스트 메소드 중 일부만 롤백을 적용하고 싶으면 메소드에 @Rollback을 부여해주면 됨.

### NotTransactional과 Propagation.NEVER

클래스 단위로 @Transactional이 지정되어 있을 때 트랜잭션이 필요하지 않은 메소드의 경우에는 @NotTransactional을 부여하면 클래스 레벨의 @Transactional 설정을 무시하고 트랜잭션을 시작하지 않은 채로 테스트를 진행.

그런데 @NotTransactional은 스프링 3.0에서 제거 대상이 됐기 때문에 사용이 꺼럼칙함.

굳이 사용하려면 사용 가능.

@NotTransactional **대신 @Tr**ansactional의 트랜잭션 전파 속성을 사용하는 방법도 존재.

@Transactional(propagation=Propagation.NEVER)

### 효과적인 DB 테스트

테스트 내에서 트랜잭션을 제어할 수 있는 네 가지 애노테이션을 잘 활용하면 DB가 사용되는 통합 테스트를 만들 때 매우 편리.

일반적으로 의존, 협력 오브젝트를 사용하지 않고 고립된 상태에서 테스트를 진행하는 단위 테스트와 DB 같은 외부의 리소스나 여러 계층의 클래스가 참여하는 통합 테스트는 아예 클래스를 구분해서 따로 만드는 게 좋음.

DB가 사용되는 통합 테스트를 별도의 클래스로 만들어둔다면 기본적으로 클래스 래벨에 @Transactinal을 부여.

DB가 사용되는 통합 테스트는 가능한 한 롤백 테스트로 만드는 것이 좋음.

테스트가 기본적으로 롤백 테스트로 되어 있다면 테스트 사이에 서로 영향을 주지 않으므로 독립적이고 자동화된 테스트로 만들기가 매우 편함.

테스트는 어떤 경우에도 서로 의존하면 안됨.

코드가 바뀌지 않는 한 어떤 순서로 진행되더라도 테스트는 일정한 결과를 내야 함.
