## 4.1 사라진 SQKException

- 예외를 처리할 때 반드시 지켜야 하는것? -> 모든 예외는 적절하게 복구되든지 아니면 작업을 중단시키고 운영자 또는 개발자에게 분명하게 통보돼야 한다.

- SQLException이 발생하는 이유
1. SQL에 문법 에러가 생김
2. DB에서 처리할 수 없을 정도로 데이터 액세스 로직에 심각한 버그가 있음
3. 서버가 죽음
4. 네트워크가 끊김 등

- throws
  API 등에서 발생하는 예외를 일일이 catch하기도 귀찮고, 큰 필요도 없고, 매번 정확하게 예외 이름을 적어서 선언하기 번거로움
  -> throws Exception이라는 모든 예외를 무조건 던져버리는 선언을 모든 메소드에 기계적으로 넣어버림
  이렇게 생긴 문제점 -> 그런 메소드 선언에서는 의미있는 정보를 얻을 수 없음.
  -> 무엇인가 실행 중에 예외적인 상황이 발생할 수 있다는 것인지, 그냥 습관적으로 복사해서 붙여놓은 것인지 알수가 없기 때문


- 예외의 종류와 특징
  - throw를 통해 발생시킬 수 있는 예외
    1. Error
        - 시스템에 뭔가 비정상 적인 상황이 발생했을 경우에 사용된다.
        - 주로 자바 VM에서 발생시키는 것이고 애플리케이션 코드에서 잡으려고 하면 안된다.
        - OutOfMemoryError나 ThreadDeath같은 에러는 catch 블록으로 잡아봤자 아무런 대응 방법이 없다
          -> 개발자 잘못이 아니라, VM 내부에서 시스템적으로 뭔가 잘못됐기 때문이다.
        - 따라서 시스템 레벨에서 특별한 작업을 하는게 아니라면 애플리케이션에서는 이런 에러에 대한 처리는 신경쓰지 않아도 된다.
    2. Exception과 체크예외
       - 개발자들이 만든 애플리케이션 코드의 작업 중에 예외상황이 발생했을 경우에 사용된다.
       - Exception클래스는
       - 1. 체크예외 -> Exception클래스의 서브클래스이면서 RuntimeException클래스를 상속하지 않은 것.
         2. 언체크 예외 -> RuntimeException을 상속한 클래스
       <img width="687" height="351" alt="{456FF913-86C1-447B-818C-158CC3AAE588}" src="https://github.com/user-attachments/assets/b749bc78-d5a0-4272-97ce-3f9cec551387" />
       - 일반적인 예외는 Exception 클래스의 서브클래스 중에서 RuntimeException을 상속하지 않은 것만을 말하는 체크예외임.
       - 체크 예외가 발생할 수 있는 메소드를 사용할 경우, 반드시 예외를 처리하는 코드를 함께 작성해야 한다.
       - 사용할 메소드가 체크 예외를 던진다면 이걸 catch문으로 잡던지, 다시 throws를 정의해서 메소드 밖으로 던져야 한다. 안하면? 컴파일 에러가 뜸
         
    3. Runtime Exception과 언체크/런타임 예외
       - 명시적인 예외처리를 강제하지 않기때문에 언체크 예외라고 불림. 개발자가 부주의해서 발생할 수 있는 경우에 생긴 예외
       - catch문으로 잡거나 throws로 선언하지 않아도 된다.
       - 런타임 예외는 주로 프로그램의 오류가 있을 때 발생하도록 의도된 것 들이다.
         - NullPointerException -> 오브젝트를 할당하지 않은 레퍼런스 변수를 사용하려고 시도했을 때 발생
         - IllegalArgumentException -> 허용되지 않는 값을 사용해서 메소드를 호출할 때 발생

    - 예외처리 방법
    1. 예외 복구
     - 예외상황을 파악하고 문제를 해결해서 정상 상태로 돌려 놓는 것
     - 예외로 인해 기본작업 흐름이 불가하면 다른 작업 흐름으로 자연스럽게 유도 (but IOException 에러 메시지가 사용자에게 그냥 던져지는건 예외복구가 아님)
     - 통제 불가능한 외부 요인으로 인해 예외가 발생하면 MAX_RETRY만큼 재시도를 하는 예시 (최대 재시도 횟수 0번)
    <img width="854" height="475" alt="image" src="https://github.com/user-attachments/assets/b58e1647-1123-436c-ad28-6099fd60ad18" />


    2. 예외처리 회피
       - 예외처리를 자신이 담당하지 않고 자신을 호출한 쪽으로 던져버리는 것.
       - throws문으로 선언해서 예외가 발생하면 알아서 던져지게 하거나/catch문으로 일단 예외를 잡은 후에 로그를 남기고 다시 예외를 던진다
       - 예외 처리를 회피하려면 반드시 다른 오브젝트나 메소드가 예외를 대신 처리할 수 있도록 던져줘야함
       <img width="1039" height="468" alt="{78F40CFB-0830-49F3-A758-5704C9277801}" src="https://github.com/user-attachments/assets/21c7371c-1b5d-4504-80a7-20bd3412932e" />
       - 예외를 회피하는 것은 예외를 복구하는 것처럼 의도가 분명해야 함 -> 콜백/템플릿처럼 긴밀한 관계에 있는 다른 오브젝트에게 예외처리 책임을 분명히 지게 하거나, 자신을 사용하는 쪽에서 예외를 다루는 게 최선의 방법이라는 분명한 확신이 있어야 한다.
      
   3. 예외전환
      - 예외 회피와 비슷하게 예외를 복구해서 정상적인 상태로는 만들 수 없기 때문에 예외를 메소드 밖으로 던지는 것
        (but 예외 회피와 달리, 발생한 예외를 그대로 넘기는게 아니라, 적절한 예외로 전환해서 던진다)



  - 예외 전환의 목적 1
         - 내부에서 발생한 예외를 그대로 던지는 것이 그 예외 상황에 대한 적절한 의미를 부여해주지 못하는 경우에 의미를 분명하게 해 줄수 있는 예외로 바꿔주기 위함. ex) API가 바생하는 기술적은 로우레벨을 상황에 적합한 의미를 가진 예외로 변경하는 것.

-> 보통, 전환하는 예외(DuplicateUserIdException)에 원래 발생한 예외(SQLException)로 만드는 것이 좋다. 
- 중첩 예외는 새로운 예외를 만들면서 생성자나 initCause()메소드로 근본 원인이 되는 예외를 넣어주면 된다.

- 예외 전환의 목적 2
 - 예외를 처라하기 쉽고 단순하게 만들기 위함 -> 포장
 - - 중첩예외를 이용해 새로운 예외를 만들고 원인이 되는 예외를 내부에 담아서 던지는 방식은 같음. but 의미를 명확하게 하려고 다른 예외로 전환하는 것은 아님. 주로 예외처리를 강제하는 체크예외를 언체크 예외인 런타임 예외로 바꾸는 경우에 사용함.

EJBException은 RUuntimeException 클래스를 상속한 런타임 예외임. 이렇게 런타임 예외로 만들어서 전달하면 EJB는 이를 시스템 익셉션으로 익식하고 트랜잭션을 자동으로 롤백해줌.

- 일반적으로 체크 예외를 계속 throws를 사용해 넘기는건 무의미함. 메소드 선언은 지저분해지고 아무런 장점이 없음.
- 어짜피 복구 불가능예외라면 가능한 한 빨리 런타임 예외로 포장해 던지게 해서 다른 계층의 메소드를 작성할 때 불필요한 throw 선언이 들어가지 않도록 해줘야함. + 예외처리 서비스 등을 이용해 자세한 로그 남기기 + 관리자에게 메일 등으로 통보 + 사용자에겐 안내 메시지

- JDBC의 한계
  - JDBC는 자바를 이요해 DB에 접근하는 방법을 추상화된 API형태로 정의해 놓고, 각 DB업체가 JDBC표준을 따라 만들어진 드라이버를 제공하게 해준다. 내부 구현은 DB마다 다르지만, JDBC의 Connection, statement, ResultSet 등의 표준 인터페이스를 통해 그 기능을 제공해주기 때문에 자바 개발자들은 표준화된 JDBC의 API에만 익숙해지면 DB의 종류에 상관없이 일관된 방법으로 프로그램을 개발 할 수 있음.

  - 비표준 SQL
    - 대부분의 DB는 표준을 따르지 않는 비표준 문법과 기능 제공
    - 호환 가능한 표준 SQL만 사용하는 방법, DB별로 별도의 DAO를 만들거나 SQL을 외부에 독립시켜서 DB에 따라 변경해 사용하는 방법이 있음
